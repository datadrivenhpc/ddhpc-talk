A Diverse Ecosystem

  • Applications build on higher-level tools and frameworks for execution and resource management:
    • Scientific workflow systems (Dispel4Py, Pegasus, ...)
    • Workload management systems (Pilot jobs, Storm, ...)
    • Distributed systems (Globus, Condor, ...)
  • Different languages and runtime environments (Python, R, C, Java, ...)
  • Dependencies / interaction with external services an data sources
  • Not just individual applications, but frameworks, etc.

Relation with HPC

Suboptimal mapping to the HPC usage model. For example:

  • Data-intensive analytics rely on predictable I/O, but there are no QoS contracts for I/O
  • Federation relies on network connectivity, but there are no QoS contracts for that either
  • Many-task applications require finegrained and flexible task placement
  • Dynamic / non-deterministic algorithms need flexible resource allocation, but the model is static

Main chanllenge: Inefficiency

  • Poor performance due to unfavorable resource mapping
  • Even worse: premature termination
    • No results = wasted CPU hours and $$$ for the user
    • "Hollow utilization" for the platform provider
    • Example: application runs out of wall-time before producing a result

Mitigate: Observe and React

Observe application behavior (intrinsic) and the behavior of the environment (extrinsic) and react accordingly, based on a set of rules:

  • to converge towards optimal efficiency
  • (or at least) prevent waste

Observe and React: Some Examples

  • If I/O degrades below a critical point, migrate affected processes or terminate application
  • If node memory is exhausted, reduce task density on the node or reduce data / array sizes
  • If CPU utilization is not saturated, increase the task density

Our Definition of 2nd Gen Apps



"Second generation HPC applications are applications that require continuous access to operational data in order to gain insights into their behavior and their environment so they can operate efficiently."

  HPC Platform Data Drought  

Platform Data Drought

HPC systems are essentially a black box (black hole?).

  • Resource management systems don't provide data:
    • Interfaces are asymmetric: control but no feedback
    • Line of sight ends at the level of job state
    • No APIs exist to introspect beyond job level
  • Platform data, if collected by admin staff, is usually not published
  • High-level statistics, historical data is usually not available programmatically

Collect Your Own!?

Let the application processes collect their own data on node / OS level while they execute.

  • How can non-system experts do this?
  • How do we get a comprehensive view of the data?
  • How do we filter relevant information from that data?
  • How do we manage at scale? 10,000 task? 100,000 tasks?
  • How do we manage data sharing archival of long-term historical data?


  Doesn't really work on application level (at scale)  

Data-Driven HPC

Apply big data principles, techniques and best practices on platform level to manage, analyse and provision operational data.

Challenge: Scale

Challenge: Scope

Different user groups and applications require different data content, frequency and resolution:

  • A non-expert users might only want a high-level warning, e.g., "I/O performance is terrible. You should abort."
  • A resource management systems might want full real-time data streams and additionally some historical data to make scheduling decissions.
